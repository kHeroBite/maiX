# kTest_run — 테스트 실행

> **프로젝트별 테스트 설정**: kTest_run_{project} 참조

## 테스트 실행 순서 (필수 — 순차 실행)

```yaml
고정_순서:
  1순위: MarsGW — 상태 확인, 로그 검증
  2순위: Mars — REST API, 로그, 스크린샷 검증
  3순위: Mars.Mobile — 설치 확인, 실행, 로그 검증

순차_실행: 앞 프로젝트 테스트 완료 후 다음 프로젝트 진행
대상_아닌_프로젝트: 스킵

실패_처리:
  해당_프로젝트: 스마트 라우팅
  나머지_프로젝트: 계속 진행 (독립적)
```

## 테스트 유형

### 1. 로그 확인

```yaml
통과_조건: ERROR/WARN/Exception 로그 0건
실패_시: 원인 분석 후 스마트 라우팅
```

### 2. REST API 테스트

```yaml
통과_조건:
  - HTTP 200 응답
  - 데이터 정확성 100%
```

### 3. 스크린샷 확인

```yaml
통과_조건:
  - UI 정상 표시
  - 텍스트 정상 렌더링
  - 컨트롤 배치 정확

UI_변경_대상_세밀_검증 (필수):
  원칙: UI 수정 작업 시 변경 대상을 스크린샷에서 육안 확인
  절차:
    1. 변경 대상 식별: 이번 작업에서 수정한 UI 요소 목록화
    2. 스크린샷 촬영: 해당 UI가 보이는 상태에서 촬영
    3. 대조 검증: 스크린샷에서 변경 대상을 직접 확인
       - 크기/위치/색상/텍스트가 계획과 일치하는지 판정
       - "전체 정상"이 아닌 "변경 대상 정상"을 구체적으로 확인
    4. 판정: PASS 시 구체적 근거 명시 (어떤 요소가 어떻게 정상인지)
  실패_시:
    UI_잘림/깨짐/불일치: kDev 복귀 (코드 수정)
    접근법_자체_오류 (3회_연속_실패): kThink (관점 전환) → kPlan 복귀
  미세_요소_확대_검증:
    대상: 아이콘, 버튼, 14px 이하 컨트롤 등 전체 스크린샷으로 식별 어려운 요소
    절차:
      1. 해당 영역의 좌표/핸들 식별
      2. 부분 스크린샷 또는 확대 캡처 API 활용
      3. 확대된 이미지에서 잘림/깨짐/비율 직접 확인
    Fallback: 부분 캡처 불가 시 → 코드 로직(좌표/크기 계산) 대조 검증으로 대체
  요구사항_기반_판정 (필수 — L-047):
    원칙: 스크린샷 PASS/FAIL은 반드시 "검증 체크리스트"와 대조하여 판정
    배경: 스크린샷을 "전체적으로 정상"이라고 판단하면 빈 공간/미배치 등 핵심 결함을 놓침
    절차:
      1. kO로부터 전달받은 "검증 체크리스트" 확인 (프롬프트에 포함)
         - 체크리스트 예시: "마지막 행 위젯 3개가 전체 너비를 빈틈없이 균등 분할"
      2. 체크리스트 항목별로 스크린샷에서 하나씩 대조
         - 각 항목에 대해 "확인됨/미확인/불일치" 명시
         - 빈 공간, 여백, 미배치 영역을 적극적으로 탐색
      3. 불일치 항목 1개라도 발견 시 → FAIL (부분 PASS 없음)
      4. 판정 결과를 구조화 출력:
         ```
         📋 스크린샷 검증 체크리스트:
         [✅] 항목1: 설명 — 확인됨 (근거)
         [❌] 항목2: 설명 — 불일치 (근거: 오른쪽 1/4 빈 공간)
         → 결과: FAIL
         ```
    검증_체크리스트_미전달_시:
      - kO 프롬프트에 체크리스트 없으면 → 사용자 요구사항 원문에서 직접 추출
      - 추출 불가 시 → "검증 기준 불명확" 경고 + 보수적 판정 (의심 시 FAIL)
    빈_공간_탐색 (레이아웃 작업 시 필수):
      - 위젯/컨트롤이 차지하지 않는 빈 영역이 있는지 적극 확인
      - "위젯이 있다" ≠ "빈 공간이 없다" — 위젯이 전체 너비를 채우는지 확인
      - 행/열 단위로 빈틈 유무를 검사
  금지:
    - 전체 스크린샷만으로 미세 요소 "정상" 판정 금지
    - 변경 대상 미확인 상태에서 PASS 선언 금지
    - 스크린샷 촬영 자체를 생략하고 커밋 금지
    - 검증 체크리스트 항목별 대조 없이 "전체적으로 정상" PASS 판정 금지 (L-047)
    - 레이아웃 작업에서 빈 공간 탐색 없이 PASS 판정 금지 (L-047)
```

### 4. 통합 테스트 (3-in-1)

```yaml
목적: REST API → 스크린샷 → 로그를 한 사이클로 검증

통과_조건 (3가지 모두):
  - REST API 응답 정상
  - 스크린샷 UI 정상
  - 로그 ERROR/WARN 0건
```

## 테스트 설정

```yaml
프로젝트별: kInfra_{project} 프로젝트스킬 참조
  - API 엔드포인트
  - 로그 경로
  - 스크린샷 경로
  - 타임아웃 기준
```

## 테스트 실패 시 라우팅 (구체적 판정)

```yaml
원칙: 테스트 실패 = 작업 미완료. 반드시 원인 분류 후 해당 단계로 복귀.

REST_API_실패:
  응답코드_비정상 (4xx/5xx):
    원인: 엔드포인트/라우팅/권한 코드 오류
    라우팅: kDev (코드 수정)

  응답값_불일치 (200이지만 데이터가 계획과 다름):
    원인_판정:
      코드_로직_오류: 비즈니스 로직/쿼리가 잘못된 결과 반환
        → kDev (해당 로직 수정)
      계획_자체_오류: 기대값이 실제 요구사항과 불일치
        → kPlan (재계획)
      DB_데이터_문제: 쿼리는 맞지만 테스트 데이터가 없음/다름
        → kDev (데이터 준비 또는 쿼리 조정)

로그_실패:
  ERROR/Exception_발생:
    원인: 런타임 오류, null reference, 타입 변환 등
    라우팅: kDev (해당 코드 수정)
  WARN_과다:
    원인: 비정상 동작 경고
    라우팅: kDev (경고 원인 제거)

스크린샷_실패:
  UI_미표시/깨짐:
    원인: 컨트롤 생성/초기화 오류
    라우팅: kDev (UI 코드 수정)
  레이아웃_불일치 (계획과 다른 배치):
    원인_판정:
      구현_오류: 코드가 계획대로 배치하지 않음
        → kDev (레이아웃 수정)
      계획_오류: 계획 자체의 레이아웃이 부적절
        → kPlan (재설계)
  데이터_불일치 (UI에 표시된 값이 계획과 다름):
    → REST API 응답값 불일치와 동일 판정 기준 적용

공통_규칙:
  - 실패 원인 불명 시 → kDebug (근본원인 분석)
  - kDev 복귀 후 수정 → 반드시 kTest_run 재실행 (스킵 금지)

동일_테스트_3회_연속_실패 (원인별 차등):
  환경_문제 (대기):
    증상: 서버 미응답, 포트 충돌, 프로세스 미시작
    대응: 환경 복구 후 재시도 (3회 카운트 리셋)
  코드_문제 (kDev 복귀):
    증상: 빌드 성공하나 런타임 오류 반복
    대응: 해당 코드만 수정 후 kTest_run 재실행
  설계_문제 (kPlan 복귀):
    증상: 3회 실패 + 코드 수정으로 해결 불가
    대응: 설계 재검토 (처음부터 재계획)
```

## 절대 금지

- 테스트 건너뛰기
- 실패 후 완료 선언
- curl --max-time 생략
- 테스트 실패를 무시하고 kDone 진입
