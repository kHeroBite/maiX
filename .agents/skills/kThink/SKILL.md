# kThink — 관점 전환

> 독립 유틸리티 스킬 — 모든 파이프라인 단계에서 호출 가능

## 역할 구분 (기존 도구와의 차별화)

| 도구 | 사고 방향 | 핵심 질문 | 트리거 |
|------|----------|----------|--------|
| sequential-thinking | 선형 전진 (A→B→C) | "다음 단계는?" | 복잡한 추론 |
| vibe-check-mcp | 반성적 검증 | "내 가정이 틀렸나?" | 편향 위험 |
| **kThink** | **전환적 탐색** | **"문제 정의 자체가 틀렸나?"** | **교착/동일 실패 반복** |

## 호출 조건

```yaml
트리거: (OR — 하나라도 해당 시)
  - 가설 2회+ 실패
  - 동일 접근법 반복 실패
  - 접근법 선택 근거 불분명
  - 수정 포인트 불명확 (복잡 시스템)

금지:
  - 교착 없이 남용 목적 호출
  - 재귀 호출 (kThink → kThink)
  - 3가지 모드 동시 실행 (1개만 선택)
```

## 모드 1: 재정의 (Reframe)

> 문제 접근 자체가 잘못된 것 같을 때

```yaml
절차:
  1. 현재 문제를 1문장으로 기술
  2. 반전:
     - "버그가 아니라 의도된 동작이라면?"
     - "호출자가 아니라 피호출자 문제라면?"
  3. 확대/축소:
     - 확대: 파일 → 모듈 → 시스템
     - 축소: 기능 → 메서드 → 변수
  4. 도메인 전환: 코드베이스 내 유사 패턴은 어떻게 해결했는가?

출력: 재정의된 문제 1문장 + 근거 1문장
```

## 모드 2: 대안 매트릭스 (Alternatives)

> 접근법 여러 개, 선택 근거 불분명할 때

```yaml
절차:
  1. 대안 최대 4개 열거 (MECE)
  2. 각각 평가:
     실현가능성(1-5) × 가치(1-5) / 위험(1-5) = 점수
  3. 판정:
     차이 >= 2.0 → 최고 점수 채택
     차이 < 2.0 → 추가 분석 (차별화 요소 탐색)

출력: 점수 테이블 + 선택 근거 1문장
```

## 모드 3: 레버리지 (Leverage)

> 복잡 시스템, 수정 포인트 불명확할 때

```yaml
절차:
  1. 관련 컴포넌트 5개 이내 나열
  2. 의존성 흐름 1줄 표현 (A → B → C)
  3. 최다 하류 영향 지점 = 레버리지 포인트

출력: "레버리지 포인트: [심볼/파일] — [기대 연쇄 효과 1문장]"
```

## 공통 규칙

```yaml
출력_제한: 500자 이내 (초과 시 압축)
모드_선택: 1개만 (3가지 전부 실행 금지)
재귀_금지: kThink → kThink 금지
남용_방지: 교착 없으면 호출 금지
```
