# kTest_quality — 품질 검증

## 실행 시점

기능 테스트(kTest_run) 통과 후 실행

## 검증팀 병렬화 (선택적)

```yaml
활성화_조건 (AND):
  - 검증 항목 3개+ (API + DB + UI 등)
  - 각 검증이 독립적 (서로 다른 대상)
  - 풀 작업 경로

비활성화: 검증 항목 2개 이하 → 리더 직접 순차 검증

에이전트_구성:
  verifier-api:
    유형: Explore (읽기 전용)
    모델: Haiku
    작업: REST API 응답 검증 + 헬스체크
    결과: scratchpad에 저장, 요약 5줄 반환

  verifier-db:
    유형: Explore (읽기 전용)
    모델: Haiku
    작업: MCP MySQL로 DB 정합성 검증
    결과: scratchpad에 저장, 요약 5줄 반환

  verifier-ui:
    유형: Explore (읽기 전용)
    모델: Haiku
    작업: 스크린샷 촬영 + UI 검증
    결과: scratchpad에 저장, 요약 5줄 반환

실행_절차:
  1. Deploy 완료 후 검증 항목 수 판단
  2. 3개+ → 에이전트 병렬 디스패치
  3. 전원 완료 대기
  4. 결과 종합 → PASS/FAIL 판정

통합_결과:
  전체_PASS: → kDone 진입
  일부_FAIL: → 실패 항목에 따라 스마트 라우팅 (위 기존 규칙 적용)
```

## 1. 요청-결과 대조 검증

```yaml
절차:
  1. 원래 사용자 요청 재확인
  2. TODO 계획 체크리스트 재확인
  3. 실제 구현 결과물과 대조
  4. 누락/불일치 항목 식별

판정:
  모든_항목_구현: PASS
  누락_있음: FAIL → kPlan 복귀
```

## 2. DB 정합성 검증 (DB 변경 시)

```yaml
절차:
  1. MCP MySQL로 SELECT 검증 (변경된 데이터 확인)
  2. mcp__mysql__get_schema로 변경된 테이블 스키마 확인
  3. FK/제약조건 확인 + 관련 테이블 동기화 확인
  4. 변경 내용이 DATABASE.md와 일치하는지 확인
  5. 한글 데이터: Python으로 별도 검증 (MCP latin1 이슈)

MCP_MySQL_규칙:
  허용: SELECT 조회, 스키마 조회(get_schema), 숫자/영문 INSERT/UPDATE
  금지: 한글 포함 INSERT/UPDATE (CLAUDE.md MCP MySQL 한글 인코딩 규칙 참조)

프로젝트별_DB_접속: kInfra_{project} 프로젝트스킬 참조

판정:
  기대값_일치: PASS
  불일치: FAIL → kPlan 복귀
```

## 3. UI/UX 품질 검증 (해당 시)

```yaml
절차:
  1. 검증 체크리스트 대조 (필수 — L-047):
     - kO로부터 전달받은 "검증 체크리스트" 항목별로 스크린샷과 대조
     - 각 항목에 PASS/FAIL + 구체적 근거 명시
     - 체크리스트 없으면 → 사용자 요구사항 원문에서 직접 추출
  2. 스크린샷 기반 레이아웃 확인
     - 빈 공간 적극 탐색: 위젯/컨트롤이 채우지 않는 빈 영역 유무 확인
     - 행/열 단위로 전체 너비 활용 여부 검사
  3. 사용자 흐름(UX) 자연스러움 확인
  4. 한글 렌더링 정상 확인
  5. 판정 결과 구조화 출력:
     ```
     📋 UI/UX 품질 검증:
     [✅] 체크리스트 항목1 — 확인됨 (근거)
     [❌] 체크리스트 항목2 — 불일치 (근거)
     [✅] 빈 공간 탐색 — 빈 영역 없음
     [✅] 한글 렌더링 — 정상
     → 결과: PASS/FAIL
     ```

판정:
  정상 (모든 체크리스트 항목 PASS): PASS
  불일치 (1개라도 FAIL): FAIL → kPlan 복귀

금지 (L-047):
  - 체크리스트 항목별 대조 없이 "UI 정상" 판정
  - 빈 공간 존재 시 "위젯은 있으므로 정상" 판정
  - 요구사항 핵심 조건(예: 빈틈없이 균등 배치) 미확인 상태에서 PASS
```

## 4. Gateway 모드 호환 검증 (해당 시)

```yaml
절차:
  1. UseGateway 분기 로직 존재 확인
  2. Gateway/직접연결 양쪽 코드 경로 확인

판정:
  양쪽_정상: PASS
  누락: FAIL → kDev 복귀
```

## 통과 조건 (모두 충족)

- 사용자 요청 모든 항목 구현됨
- TODO 체크리스트 100% 달성
- DB 변경 시 정합성 확인됨
- UI 변경 시 스크린샷 정상
- Gateway 양쪽 모드 동작 확인 (해당 시)

## 실패 시 라우팅 (경중 구분)

```yaml
실패_시_라우팅:
  품질_미달 (코드 수준): kDev 복귀 (수정 후 재검증)
    - UI 미세 조정 (폰트, 정렬, 색상)
    - 데이터 검증 미흡 (조건 추가만 필요)
    - 누락된 예외 처리 추가

  설계_결함 (아키텍처): kPlan 복귀 (재설계)
    - 요구사항 미충족 (설계 재검토 필요)
    - 데이터 정합성 오류 (쿼리/로직 전면 재설계)
    - 아키텍처 문제 (구조적 변경 필요)

  Gateway_누락: kDev 복귀 (Gateway 분기 추가)
    - Gateway 모드 분기 누락
    - UseGateway 조건 처리 미흡

  기본값: kDev 복귀 (kPlan은 설계 결함 시에만)

공통:
  1. 실패 원인 및 증거 기록
  2. TODO 파일에 실패 사유 추가
```
