# kDebug — 체계적 디버깅

> 독립 유틸리티 스킬 — 모든 파이프라인 단계에서 호출 가능

## Quick Debug 분기 (명백한 단순 오류)

```yaml
Quick_Debug_조건: (AND — 모두 충족 시)
  - 오류 원인이 명백 (오타, 상수 오류, off-by-one 등)
  - 단일 파일 수정으로 해결 가능
  - 사이드 이펙트 없음

Quick_Debug_절차:
  - Phase 1 스킵 (조사 불필요 — 원인 자명)
  - Phase 2: 패턴 매칭만 (30초)
  - Phase 3 스킵 (가설 불필요)
  - Phase 4: 즉시 수정
```

## 경량 모드 (간단 버그)

```yaml
경량_모드_조건:
  - 원인이 명백 (오타, 상수 오류, 단순 로직 실수)
  - Quick Debug보다는 복잡하나 Phase 2-3이 불필요한 수준

경량_모드_절차:
  Phase 1(조사) → Phase 4(수정) 직행 (Phase 2-3 생략)

판단_기준:
  원인이 30초 내 식별되면 경량 모드 적용
  30초 초과 시 Standard Debug(4 Phase) 전환
```

> Quick Debug / 경량 모드 조건을 충족하지 않으면 아래 Standard Debug(4 Phase)를 따릅니다.

## 철칙 (Standard Debug)

```
근본 원인 조사 없이 수정 금지
Phase 1 완료 전 수정 제안 불가
```

## Phase 1: 근본 원인 조사

**모든 수정 전 필수:**

1. **에러 메시지 주의 깊게 읽기**
   - 스택 트레이스 완전히 읽기
   - 줄 번호, 파일 경로, 에러 코드 기록
   - 건너뛰기 금지

2. **일관되게 재현**
   - 정확한 트리거 단계 확인
   - 재현 불가 → 더 많은 데이터 수집 (추측 금지)

3. **최근 변경사항 확인**
   - Git diff, 최근 커밋
   - 새 의존성, 설정 변경, 환경 차이

4. **다중 컴포넌트 증거 수집**
   - 각 컴포넌트 경계마다 진입/이탈 데이터 로그
   - 환경/설정 전파 검증
   - 한 번 실행하여 실패 컴포넌트 식별

5. **데이터 플로우 추적**
   - 나쁜 값의 발생 위치 → 위로 추적
   - 증상이 아닌 소스에서 수정

## Phase 2: 패턴 분석

1. **작동하는 예시 찾기** — 코드베이스에서 유사한 작동 코드
2. **참조와 비교** — 모든 줄 읽기, 훑어보기 금지
3. **차이점 식별** — 아무리 작아도 모든 차이 나열
4. **의존성 이해** — 필요한 컴포넌트, 설정, 가정

## Phase 3: 가설 및 테스팅

0. **문제 재정의 체크** (2회+ 가설 실패 시 필수):
   - 현재 정의가 올바른가?
   - 반대 관점: 버그가 아니라 설계 의도라면?
   - 범위 전환: 이 파일이 아니라 호출자/피호출자 문제라면?
   - 교착 시: /kThink 재정의 모드

1. **단일 가설 형성**: "Y 때문에 X가 근본 원인"
2. **최소한으로 테스트**: 한 번에 하나 변수만
3. **계속 전 검증**: 통함 → Phase 4, 안 통함 → 새 가설
4. **모를 때**: "이해 못함" 말하기, 아는 척 금지

## Phase 4: 구현

1. **실패 테스트 케이스 생성** (수정 전 필수)
2. **단일 수정 구현** — 한 번에 하나, "이왕 여기 왔으니" 금지
3. **수정 검증** — 테스트 통과, 다른 테스트 깨짐 없음

### 3회 실패 규칙

```yaml
수정_시도_카운트:
  < 3회: Phase 1로 복귀, 새 정보로 재분석
  == 3회 (순서 필수):
    1단계_Codex_리뷰: Codex MCP에 코드 리뷰 요청 (아래 Codex 리뷰 섹션 참조)
      → Codex 가이드로 해결: Phase 4 재진입 (수정 적용)
      → Codex 가이드 불충분: 2단계로
    2단계_kThink: /kThink 레버리지 모드 (최소 변경 최대 효과 지점 탐색)
      → 레버리지 포인트 발견: 해당 지점에서 Phase 1 재시작
      → 미발견: /kPlan 복귀 (전체 재설계)
  > 3회: 절대 금지

아키텍처_문제_신호:
  - 각 수정이 다른 곳에 새 문제 드러냄
  - "대규모 리팩토링" 필요
  - 각 수정이 새 증상 생성
```

### Codex 코드 리뷰 (반복 실패 시 자동 발동)

> 공식 OpenAI Codex MCP(`codex`, `codex-reply`)를 활용한 외부 코드 리뷰.
> 동일 문제가 반복 실패 시, kThink 전에 먼저 Codex에게 가이드를 요청한다.

```yaml
발동_조건 (OR — 하나라도 해당 시 자동 발동):
  A. kDebug_내_3회_실패: Phase 4 수정 시도 3회 실패 (기존 3회 실패 규칙)
  B. kTest_kDev_3회_왕복: kTest 실패 → kDev 수정 → kTest 실패 사이클이 3회 반복
     (파이프라인 레벨 감지 — kTest에서 동일 오류로 kDev에 3회 복귀)
  C. 사용자_피드백_3회_미해결: 사용자가 동일 문제를 3회 이상 지적했으나 해결 못함
     ("안 되잖아", "또 같은 문제", "이거 아직도 안 고쳐졌네" 등)

절차:
  1. codex 도구 호출:
     프롬프트: |
       다음 코드에서 반복적으로 해결 실패 중인 문제를 분석하라.
       {A: 3회 수정 시도 / B: kTest-kDev 3회 왕복 / C: 사용자 3회 지적}했으나 모두 실패했다.
       근본 원인과 구체적 수정 방향을 제시하라.
     입력:
       - 문제 파일 코드 (관련 부분)
       - 에러 메시지 / 증상 설명
       - 시도한 수정 내역과 각 실패 이유
  2. Codex 피드백 수신
  3. 피드백 평가:
     유효: Codex 가이드 기반으로 Phase 4 재진입
     불충분: kThink 레버리지 모드로 전환 (기존 흐름)
  4. codex-reply로 후속 질문 가능 (최대 2회)

출력:
  발동_시: "🔍 반복 실패 ({A/B/C}) — Codex 코드 리뷰 요청 중..."
  완료_시: "✅ Codex 리뷰 완료 — {가이드 유효/불충분}"

스킵_조건:
  - Codex MCP 미설치/연결 불가 → 즉시 kThink로 전환
  - Codex 응답 지연 10초+ → 스킵, kThink로 전환

주의:
  - Codex 피드백은 참고용 — 맹목적 적용 금지
  - Codex 가이드 적용 후에도 실패 시 → kThink (1단계 실패 = 2단계 진행)
  - 동일 문제에 대해 Codex 리뷰는 1회만 발동 (무한 루프 방지)
```

### Codex 기술 자문 (수동 호출 가능)

> 디버깅 외에도 기술적 판단이 필요할 때 Codex에 자문을 요청할 수 있다.

```yaml
발동_조건 (수동 — 아래 상황에서 Claude가 자율 판단):
  - 라이브러리/프레임워크 특정 동작이 불확실할 때
  - 성능 최적화 방향 결정이 어려울 때
  - 아키텍처 패턴 선택에서 트레이드오프 비교가 필요할 때
  - 사용자가 "다른 의견도 들어봐", "Codex에 물어봐" 등 명시적 요청

절차:
  1. codex 도구 호출:
     프롬프트: 구체적 기술 질문 + 컨텍스트 (코드/환경/제약)
  2. 피드백을 Claude가 평가 후 반영 여부 결정
  3. codex-reply로 후속 질문 (필요 시 최대 2회)

원칙:
  - 자율 발동은 디버깅 교착 시에만, 그 외에는 사용자 요청 시만
  - 일상적 코딩에는 사용하지 않음 (이중 비용 방지)
```

## Red Flags — 중단하고 Phase 1 복귀

- "빠른 수정 하나만"
- "X 변경 시도하고 통하는지 보자"
- "여러 변경 추가 후 테스트"
- "아마도 X, 그거 고쳐보자"
- "완전히 이해 못하지만 통할지도"
- "한 번 더 수정 시도" (2회+ 실패 후)

## 합리화 방지

| 변명 | 현실 |
|------|------|
| "이슈 간단해" | 간단한 버그에도 프로세스는 빠름 |
| "응급이라 시간 없어" | 체계적 디버깅이 추측보다 빠름 |
| "한 번에 여러 수정" | 무엇이 통했는지 격리 불가 |
| "참조 너무 길어" | 부분 이해는 버그 보장 |
| "한 번 더 수정 시도" (3회+) | 아키텍처 문제, 패턴 질문 |

## 사용자 리다이렉션 신호 (즉시 Phase 1 복귀)

```yaml
신호_감지:
  - "그건 아닌 것 같은데?", "안 되잖아", "Stop guessing"
  - "다시 처음부터", "Ultrathink this", "왜 같은 시도?"
  - "다른 방법 없어?", "Is that not happening?"
  - "We're stuck?", "Will it show us...?"

대응:
  즉시_Phase1_복귀: 모든 가설 폐기 → Root Cause 재분석
  금지: 동일 접근법 반복, "한 번 더 시도"
  규칙_위반: 형식적 절차 준수만으로 부족 — 디버깅 정신까지 준수
```

## Quick Reference

| Phase | 핵심 활동 | 성공 기준 |
|-------|----------|----------|
| 1 Root Cause | 로그/코드/데이터 수집 | 원인 1개 확정 |
| 2 Pattern | 유사 사례 검색 | 패턴 식별됨 |
| 3 Hypothesis | 가설 수립+검증 | 가설 통과 |
| 4 Implementation | 코드 수정+테스트 | Red→Green 확인 |
